ACID Guarantees — tap each to expand
A Atomicity ▼
All operations in the transaction succeed, or none take effect. The debit and credit are an indivisible pair — if either fails, the database undoes both automatically.
C Consistency ▼
Every committed transaction leaves the database in a valid state. Constraints like "balance ≥ 0" or foreign-key rules are enforced — a violating transaction is rejected in full, never partially.
I Isolation ▼
Concurrent transactions cannot see each other's uncommitted changes. Alice's pending debit is invisible to every other connection until COMMIT. (Explored in depth in the next lesson.)
D Durability ▼
Once the database confirms COMMIT, the changes are written to durable storage. They will survive a crash or power loss — a committed transaction is permanent from that moment on.
