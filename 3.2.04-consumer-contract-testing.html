Step 01 — The Change

A developer renames a response field

A service returns user records. The developer renames status to state for clarity. Internal logic is unchanged — only the field name moves.

Before
{ "user_id": "u_42", "status": "active", "email": "jo@co.com" }
After
{ "user_id": "u_42", "state": "active", "email": "jo@co.com" }
The change looks clean and intentional. No business logic changed. Is this safe to ship?
Step 02 — Unit Tests

The developer runs the unit test suite

These tests were written to check the service's own internal logic. They pass when the code does what the developer intended.

✓
returns 200 for a valid user ID PASS
✓
returns 404 for an unknown user ID PASS
✓
"state" field is a non-empty string PASS
3 / 3 passed
Green unit tests confirm the code does what you intended. But they were written by you, for you — never from a consumer's point of view.
Step 03 — Consumer Contract Suite

The contract suite finds a broken promise

The contract suite was written from Consumer A's point of view. It asserts only the promises Consumer A actually depends on — not the provider's internal behavior.

✗
Consumer A · response includes field "status" FAIL
✓
Consumer A · response includes field "email" PASS
✓
Consumer A · 404 returned for unknown user PASS
1 failure — a consumer promise was broken
Consumer A expected: { "status": string } Received: { "state": "active" } → Consumer A reads the "status" field at runtime. It will crash in production when this change ships.
Step 04 — Your Decision

The contract is red. What do you do?

Pick every response that correctly resolves the broken contract. One of the three options is always wrong — no matter the situation.
