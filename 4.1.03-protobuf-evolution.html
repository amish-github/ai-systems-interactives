Step through to see how schema changes travel on the wire
Step 1 of 5
The field number is the contract

Every Protobuf field has a name, a type, and a number. On the wire, the name is never sent. Only the number travels between systems.

message User {
  string  email        = 1;
  string  display_name  = 2;
  int32   age          = 3;
}
What the decoder actually sees on the wire:
tag=1
email
·
tag=2
display_name
·
tag=3
age
The decoder reads tag=1 and knows it means email. It never sees the word "email" in the bytes. This means renaming a field is harmless, but changing a number breaks every client that relied on the old one.
Step 2 of 5
Safe change: adding a new field

To add phone_number, pick the next unused number and leave all existing numbers untouched.

message User {
  string  email        = 1;
  string  display_name  = 2;
  int32   age          = 3;
  string  phone_number  = 4;  // ← new
}
Wire — v2 client now sends:
tag=1
email
·
tag=2
display_name
·
tag=3
age
·
tag=4
phone_number
tag=4 has never existed before, so no existing client has ever sent it. Old clients keep sending tags 1–3 and work without any change. This is how a Protobuf schema grows safely.

Now the new client is deployed before all servers are updated. A new client sends tag 4. The old server has never heard of it. What happens?

Step 5 of 5
Breaking change: reusing a retired tag number

Someone removes age and places a new role field at the same number to "avoid a gap." Old clients still send age data as tag=3.

message User {
  string  email        = 1;
  string  display_name  = 2;
  string  role          = 3;  // ← reused!
}
💥
Silent data corruption
Old clients send age = 42 as tag=3. The new server reads that as role and tries to decode a 32-bit integer as a string. No error is raised — just wrong data flowing silently through the system.
✗ Data silently corrupted
The fix: use reserved when removing a field. This tells the compiler to reject any future field that tries to claim that number or name.
message User {
  reserved 3;  reserved "age";  // locked forever
  string  email        = 1;
  string  display_name  = 2;
  string  role          = 4;  // safe new number
}
