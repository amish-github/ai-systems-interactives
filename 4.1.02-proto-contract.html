One file, two jobs
A .proto file defines two things: the shape of data (messages) and the operations a server offers (a service). Both the client and the server generate their code from this single file — not from separate descriptions that have to stay in sync.
orders.proto — single source of truth
syntax = "proto3";
package orders.v1;

// ① Messages — the shape of data
message Order {
  string id = 1;
  string customer_id = 2;
  OrderStatus status = 3;
}

// ② Service — operations the server offers
service OrderService {
  rpc GetOrder(GetOrderRequest) returns (GetOrderResponse);
}
This one file is the contract. Neither side writes its own description and hopes the two match — both generate from the same source of truth.

Human-readable; used in generated code. The name is not sent over the wire — so you can rename it without breaking encoded messages.

Field tag
The permanent numeric ID used in the binary encoding. This IS sent over the wire. Never reassign a tag to a different field — doing so silently corrupts messages.

tag — travels on the wire
The tag number — not the name — is what gets transmitted. That is part of why proto encoding is compact. It is also why reassigning a tag to a different field silently corrupts messages already in flight.
Run protoc: one file fans out to both sides
You run the Protocol Buffers compiler (protoc) once. It reads the proto file and emits typed source code for both the client and the server in your language of choice.

Because both sides come from the same file, the compiler in each language enforces the contract. A type mismatch or missing field is caught at compile time — not at runtime in production.
What the generated code looks like
You do not write this code. The compiler writes it for you. Both sides get the exact same types — derived from the single proto file.

    self,

  ) -> GetOrderResponse:
    # calls server over gRPC
    ...
Server (generated)
# Auto-generated base class

  def get_order(
    self,
    req: GetOrderRequest
  ) -> GetOrderResponse:
    raise NotImplementedError
Both sides use the identical GetOrderRequest and GetOrderResponse types — generated from the same proto definitions. There is no way for one side to drift from the other silently.
Enforced at compile time, not at runtime
The proto file is not documentation that sits beside your code hoping to stay accurate. It is the real input to the compiler — and that single fact changes when mismatches are caught.
JSON / REST (conventional)
The agreement lives in a README, a doc, or people's memory. Client and server each write their own description and hope the two match. Mismatches surface as errors at runtime — in production, under real traffic.
gRPC + Protobuf (enforced)
Both sides generate from a single proto file. The compiler in each language checks the types. A missing field or wrong type is caught at compile time — before any code ships.
When you change a proto file you are changing the real agreement between every client and every server. Treat it with the same care you would give to any shared interface.
