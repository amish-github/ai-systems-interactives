Where do you start when designing a schema?
The feature: An e-commerce app. The most common screen loads "all orders for a given customer, newest first."

You are about to create the customers and orders tables. What should drive the first design choice?

Option A
Model every entity in the abstract — customers, orders, products, shipping addresses, promotions — and build a complete picture before thinking about any specific query.
Option B
Write down the key access pattern — "orders for a customer, newest first" — and shape the tables so that lookup is cheap.
Not quite. Modeling everything abstractly first is the habit this lesson warns against. It produces tables that are awkward for the queries that actually run, because no one asked what the schema would be used for.
Exactly. The access pattern "orders for a customer, newest first" immediately tells you the orders table needs a customer column and that lookup must be indexed. You only know that by looking at the real query first.
Decision 2 of 4
What should the primary key of customers be?
The access pattern: Customers log in with their email. The orders table will need to reference each customer.
Natural key
CREATE TABLE customers ( email TEXT PRIMARY KEY, created_at TIMESTAMPTZ );
Email is unique — use it as the identifier.
Surrogate key
CREATE TABLE customers ( id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, email TEXT NOT NULL UNIQUE, created_at TIMESTAMPTZ );
Generate a stable ID. Store email separately.
Risky choice. Email addresses change. Once the orders table stores email as a foreign key, updating one customer's email means updating that value in every order row that references it. A surrogate key never changes because it has no real-world meaning to update.
Right. The generated id never changes. Other tables reference id, so a customer can update their email without cascading rewrites across every row that points to them.
Decision 3 of 4
Where does the link between orders and customers go?
The access pattern: "Load all orders for a given customer." One customer can have many orders.
Option A
-- customers stores -- an array of order IDs CREATE TABLE customers ( id BIGINT PRIMARY KEY, order_ids BIGINT[] );
The "one" side holds a list pointing to the "many" side.
Option B
-- each order points back -- to its customer CREATE TABLE orders ( id BIGINT PRIMARY KEY, customer_id BIGINT REFERENCES customers(id), placed_at TIMESTAMPTZ );
The "many" side carries the foreign key.
Not quite. Storing an array of IDs on the parent breaks relational modeling and makes queries awkward. In a relational database, the "many" side carries the foreign key — each orders row stores the customer_id it belongs to.
Correct. Each order row stores a customer_id foreign key. To answer "all orders for customer 42," the database looks for rows where customer_id = 42 — exactly what the access pattern requires.
